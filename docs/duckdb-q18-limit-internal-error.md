# DuckDB internal error while planning LDBC BI Query 18 with `LIMIT`

## Summary

LDBC SNB BI Query 18 triggers an internal DuckDB exception when its grouped
anti-edge query ends in `LIMIT 20`:

```text
Invalid Error: unordered_map::at: key not found
```

The query succeeds and returns the expected empty result when `LIMIT 20` is
removed. It also succeeds with the optimizer disabled. This should be reported
as an internal optimizer error: a valid SQL query should either return a result
or produce a normal SQL error, not expose a C++ container exception.

- Originally tested DuckDB source ID: `c523bc547f`
- Originally tested DuckDB commit:
  `c523bc547f590dc03526d9d5262e289f64aa5c60`
- Latest `main` verification on 2026-07-14: `v1.6.0-dev10659`
- Latest `main` source ID: `7eaa6a2ffb`
- Latest `main` commit: `7eaa6a2ffbc699ff366026860192c27895ff8861`
- Extension commit: `98ea459dcebaf46ad2974d3c97ad202c877a149f`
  plus the uncommitted BI-query implementation under review
- Platform: macOS Darwin 25.5.0, arm64
- Dataset: deterministic LDBC SNB BI SF0.003 generated by this extension

The minimal one-row reproducer fails identically on both DuckDB revisions. On
latest `main`, the failing command exits with status 1 and the no-`LIMIT`
control exits with status 0 and returns zero rows.

## Reproducer

A one-row standalone reproducer that does not require the extension is in
[`reproducers/duckdb-q18-limit-internal-error`](../reproducers/duckdb-q18-limit-internal-error/README.md).
That is the preferred reproducer for an upstream issue.

The following extension-backed reproducer records how the problem was first
isolated from the LDBC dataset.

Build the extension and open a fresh database from the repository root:

```sh
make release
build/release/duckdb /tmp/duckdb-q18-limit-repro.duckdb
```

Paste the following SQL into the DuckDB shell. The first statement creates the
small deterministic dataset; generation takes roughly one to two seconds on the
machine listed above.

```sql
CALL ldbcgen(sf := 0.003, schema := 'q18_repro');

WITH RECURSIVE
Person_hasInterest_Tag AS (
    SELECT * FROM q18_repro.Person_hasInterest_Tag
),
Tag AS (
    SELECT * FROM q18_repro.Tag
),
Person_knows_Source AS (
    SELECT * FROM q18_repro.Person_knows_Person
),
Person_knows_Person AS (
    SELECT creationDate, Person1Id, Person2Id
    FROM Person_knows_Source

    UNION ALL

    SELECT creationDate, Person2Id, Person1Id
    FROM Person_knows_Source
),
PersonWithInterest AS (
    SELECT pt.PersonId
    FROM Person_hasInterest_Tag pt, Tag t
    WHERE t.name = 'Frank_Sinatra'
      AND pt.TagId = t.id
),
FriendsOfInterested AS (
    SELECT k.Person1Id AS InterestedId,
           k.Person2Id AS FriendId
    FROM PersonWithInterest p, Person_knows_Person k
    WHERE p.PersonId = k.Person1Id
)
SELECT k1.InterestedId,
       k2.InterestedId,
       count(k1.FriendId) AS mutualFriendCount
FROM FriendsOfInterested k1
JOIN FriendsOfInterested k2
  ON k1.FriendId = k2.FriendId
WHERE k1.InterestedId <> k2.InterestedId
  AND (k2.InterestedId, k1.InterestedId) NOT IN (
      SELECT Person1Id, Person2Id
      FROM Person_knows_Person
  )
GROUP BY k1.InterestedId, k2.InterestedId
ORDER BY mutualFriendCount DESC,
         k1.InterestedId,
         k2.InterestedId
LIMIT 20;
```

### Actual result

```text
Invalid Error: unordered_map::at: key not found
```

The DuckDB CLI exits with status 1.

### Expected result

The selected tag has no matching recommendation pattern at SF0.003, so the
query should succeed with these columns and zero rows:

```text
InterestedId  InterestedId  mutualFriendCount
0 rows
```

## Controls and narrowing

The following changes were tested against a fresh generated database:

| Variation | Result |
|---|---|
| Query exactly as shown | Internal `unordered_map::at` exception |
| Remove `LIMIT 20` | Succeeds, zero rows |
| Keep `ORDER BY`, remove only `LIMIT 20` | Succeeds, zero rows |
| Add `PRAGMA disable_optimizer` before the failing query | Succeeds, zero rows |
| Disable only `cte_inlining` | Succeeds, zero rows |
| Disable only `common_subplan` | Succeeds, zero rows |
| Disable only `join_order` | Succeeds, zero rows |
| Disable only `top_n` | Internal exception remains |
| Disable only `limit_pushdown` | Internal exception remains |

The individual optimizer results suggest an interaction between CTE reuse,
join planning, and the limiting plan. They do not establish which optimizer is
the root cause.

For example, this control succeeds:

```sql
SET disabled_optimizers = 'cte_inlining';
-- Run the failing SELECT above again.
```

## Query 18 background

The query finds pairs of people interested in a given tag who share friends but
do not already know each other. The official reference SQL expresses the
negative edge with a correlated `NOT EXISTS`. During reduction, replacing that
predicate with the equivalent non-null row-value `NOT IN` still produced the
internal error whenever `LIMIT 20` was present, yielding the reproducer above.

`Person1Id` and `Person2Id` are non-null in the generated relationship, so the
two anti-edge formulations are equivalent for this dataset.

## Extension workaround

The extension currently avoids the failing limiting plan by using a window
filter:

```sql
GROUP BY k1.InterestedId, k2.InterestedId
QUALIFY row_number() OVER (
  ORDER BY count(k1.FriendId) DESC,
           k1.InterestedId,
           k2.InterestedId
) <= 20
ORDER BY mutualFriendCount DESC,
         k1.InterestedId,
         k2.InterestedId;
```

This preserves the reference query's deterministic top-20 ordering. It was
also checked with the denser `Elizabeth_II` parameter: the workaround returned
the same 12 ordered rows as the reference anti-edge query executed without the
failing `LIMIT`.

The implemented workaround is in
[`src/queries/bi-18.sql`](../src/queries/bi-18.sql), and its result is covered by
[`test/sql/ldbc_bi_results.test`](../test/sql/ldbc_bi_results.test).
